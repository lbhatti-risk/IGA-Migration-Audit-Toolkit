# IPE Integrity Validation Checklist — Automated IGA and PAM Environments

IPE (Information Produced by the Entity) risk is the primary audit risk in automated Identity Governance environments. When a UAR population is generated by an automated query rather than a manual extract, the auditor must validate that the query logic is complete, accurate, and has not been manipulated — before placing any reliance on the output.

This checklist provides a structured methodology for IPE validation during IGA and PAM audits. All scenarios are synthetic.

---

## What is IPE Risk in an IGA/PAM Context?

In a manual UAR environment, IPE risk is limited: a human exports a list, and the auditor tests the export.

In an automated environment (e.g., Veza, SailPoint, CyberArk reports), the risk shifts:

- The query may not capture all privileged role categories
- Custom or non-standard roles may fall outside the tool's default polling scope
- Mid-period platform changes may alter which accounts the query returns
- The individual who configured the query may hold privileged access themselves

The auditor's job is to validate the *query*, not just the *output*.

---

## Phase 1 — Pre-Testing: Establish the Control Design

| # | Step | Evidence Required | Pass Criteria |
|---|------|-------------------|---------------|
| 1.1 | Obtain the prior year (PY) control design for the UAR | PY workpaper or RACM | UAR scope and privileged categories clearly defined |
| 1.2 | Confirm current year (CY) control design matches PY, or document changes | CY walkthrough notes; RACM update | Any design changes are documented and approved |
| 1.3 | Identify all privileged role categories that must be included in the UAR population | Control design document; AD group structure | Complete list of privileged categories confirmed with client |
| 1.4 | Confirm which automated tool generates the UAR population | Walkthrough; system documentation | Tool confirmed (e.g., Veza, CyberArk, SailPoint) |
| 1.5 | Identify who configured the automated query | System admin logs; IT team confirmation | Confirm query owner does not hold privileged access in scope |

---

## Phase 2 — Query Logic Validation

| # | Step | Evidence Required | Pass Criteria |
|---|------|-------------------|---------------|
| 2.1 | Obtain the query configuration or report settings from the automated tool | Screenshot or export of query parameters | Query is documented and version-controlled |
| 2.2 | Confirm the query covers all privileged role categories identified in Phase 1 | Query output mapped against Phase 1 category list | All categories present; no gaps |
| 2.3 | Test for non-standard or custom roles that may fall outside default polling | AD group listing; platform role catalogue | All custom roles confirmed in or out of scope with rationale |
| 2.4 | Confirm the query runs against the live directory (not a cached or stale dataset) | System architecture confirmation; query timestamp | Output timestamp aligns with UAR period |
| 2.5 | Where a mid-period platform change occurred, confirm query validity for each sub-period | Change log; dual query outputs | Separate query outputs obtained and reconciled for each period |

---

## Phase 3 — Population Completeness Reconciliation

This is the core IPE test. The auditor independently extracts the privileged user population from the authoritative source and reconciles it against the automated tool's output.

| # | Step | Evidence Required | Pass Criteria |
|---|------|-------------------|---------------|
| 3.1 | Obtain an independent extract of all privileged accounts from the authoritative source (e.g., Active Directory, HR system) | AD admin extract or HR feed export | Extract obtained independently of the UAR tool |
| 3.2 | For each privileged category, cross-reference the tool output against the AD extract | Side-by-side comparison workpaper | Zero delta, or all deltas explained and documented |
| 3.3 | Confirm any accounts in the AD extract but absent from the tool output | Exception log | Exceptions investigated — confirm whether excluded by design or omitted in error |
| 3.4 | Confirm any accounts in the tool output but absent from the AD extract | Exception log | Stale or orphaned accounts identified and reported |
| 3.5 | Where third-party or MSP accounts exist, confirm they appear in both extracts | Third-party account list; provisioning records | No third-party accounts excluded from UAR population |

---

## Phase 4 — Mid-Period Migration Testing

Applies where the UAR process changed during the audit period (e.g., from manual Excel to an automated tool).

| # | Step | Evidence Required | Pass Criteria |
|---|------|-------------------|---------------|
| 4.1 | Confirm the exact date the process changed | Change management record; email confirmation | Transition date documented |
| 4.2 | Obtain the manual UAR evidence for the pre-transition period | Excel extract or manual log | Manual UAR covers all accounts active before transition date |
| 4.3 | Obtain the automated tool output for the post-transition period | Tool report for post-transition period | Automated UAR covers all accounts from transition date |
| 4.4 | Confirm the two populations align at the transition point | Reconciliation workpaper | Accounts active at transition date appear in both periods |
| 4.5 | Obtain evidence of a parallel run or reconciliation at the time of migration | Migration log; parallel output | Formal reconciliation performed; exceptions resolved |

---

## Phase 5 — Exception Handling and Reporting

| # | Step | Evidence Required | Pass Criteria |
|---|------|-------------------|---------------|
| 5.1 | For any IPE exceptions identified, obtain management's explanation | Management response email or meeting notes | Explanation is documented and plausible |
| 5.2 | Assess whether exceptions are isolated or indicate a systemic query flaw | Pattern analysis across exception log | Systemic issues escalated before client communication |
| 5.3 | Where query completeness cannot be confirmed, escalate to senior management | Internal escalation email or meeting note | Escalation documented before issuing any finding |
| 5.4 | Document IPE conclusion in workpaper | IPE conclusion paragraph | Conclusion states reliance position clearly: full, partial, or no reliance |

---

## IPE Conclusion Language Templates

**Full reliance:**
> The automated query has been validated against an independent AD extract for all [X] privileged categories defined in the control design. Zero exceptions were identified. Full reliance is placed on the tool output as the UAR population.

**Partial reliance (mid-period migration):**
> The UAR population covers two sub-periods: [dates] via manual process and [dates] via automated tool. Each has been independently validated. Reliance is placed on both outputs for their respective periods.

**No reliance / exception:**
> The automated query was found to exclude [category]. An independent AD extract identified [N] accounts not captured in the tool output. A population completeness exception has been raised. No reliance is placed on the tool output pending management remediation.

## Common Findings Reference

| Finding | Root Cause | Typical Severity |
|---------|------------|-----------------|
| Privileged category absent from query | Query configured against a subset of AD groups; custom roles not added | High |
| Third-party accounts excluded | MSP accounts in a separate OU not included in query scope | High |
| Sole reviewer holds privileged access | Control design does not prohibit reviewer from reviewing own access | High |
| No parallel run at migration | Transition from manual to automated process not formally reconciled | Medium |
| Stale accounts in tool output | Deprovisioning lag between HR termination and AD removal | Medium |
| Query config access not restricted | Platform admin who configures query also holds privileged access | Medium |

## Regulatory Alignment

| Regulation / Framework | Relevant Requirement |
|------------------------|----------------------|
| SOX / ITGC | IPE must be validated before reliance is placed on system-generated reports used as key controls |
| DORA (Art. 9, 10) | Accuracy and completeness of ICT access records required; automated controls subject to equivalent validation |
| ISO 27001:2022 (A.8.2, A.8.5) | Privileged access rights shall be allocated on a need-to-use basis; access reviewed at regular intervals |
| PCAOB AS 2201 | Auditor must evaluate completeness and accuracy of data produced by the entity used in testing |
